# PRD Learning Notes

## What is a PRD?
A Product Requirements Document defines WHAT to build before HOW to build it.

## Why PRD Before Code?
- Prevents building wrong features
- Aligns team on scope
- Used in every tech company (Google calls it "Design Doc", Amazon "PR/FAQ")

## Key Concepts Learned

### MoSCoW Method
- **M**ust have - MVP dies without it
- **S**hould have - Important but not critical
- **C**ould have - Nice to have
- **W**on't have - Explicitly excluded

### Persona-Based Development
Always ask: "Who is using this and what's their pain point?"

### MVP Mindset
"Minimum Viable Product" = smallest thing that delivers value

## Common Interview Questions

**Q: How do you scope an MVP?**
A: Start with user pain points, identify must-have features that solve that pain, cut everything else. Use MoSCoW framework. Set measurable success criteria.

**Q: What's the difference between PRD and System Design?**
A: PRD = WHAT (features, users, goals). System Design = HOW (architecture, databases, APIs).

**Q: How do you handle scope creep?**
A: Point to the PRD. If a new feature is requested, ask: "Does this serve our MVP goal? If yes, what existing feature should we trade off?"

## Startup Relevance
- Investors expect clear MVP scope
- Shows you can think beyond code
